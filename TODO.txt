# exo_kernels.py Review

## Bugs

1. **rmsnorm_bwd:112** - The `+ dx[i, j]` term conflates two operations.
   - Current: `out[i, j] = out[i, j] * scale - x_pre[i, j] * corr + dx[i, j]`
   - The rmsnorm backward formula is correct: `out * scale - x_pre * corr`
   - But `+ dx` adds a residual gradient, mixing concerns
   - This works for the current architecture (residual connections) but is non-standard
   - Consider: rename to `rmsnorm_bwd_add` or split into two ops

## Performance Issues

2. **cross_entropy:150-152** - O(N) search for target token is inefficient.
   - Current: Loops through all vocab indices to find target
   - Fix: Direct index `p_target = probs[t, target_ids[t]]`
   - Exo may not support dynamic indexing cleanly, investigate.

3. **cross_entropy_bwd:162-165** - Same O(N) conditional pattern.
   - Conditional inside tight loop is bad for hardware acceleration.
   - Consider restructuring or using masked operations.

## Style/Correctness Concerns

4. **select() semantics** - Verify 4-argument usage.
   - Pattern: `select(a, b, val_if_a_gt_b, val_if_b_gt_a)` for max?
   - Used in softmax:59-61, relu:75, relu_bwd:82
   - Confirm this matches exo's actual select function signature.

5. **adam beta values** - Hardcoded and non-standard.
   - Current: beta1=0.85, beta2=0.99
   - Standard: beta1=0.9, beta2=0.999
   - Consider making these parameters or documenting the choice.

6. **softmax initialization** - mx = -1e10 may not be safe for all inputs.
   - Better: Use -inf or first element of the row.
   - f64 can represent values much smaller than -1e10.

## Potential Improvements

7. **Memory annotations** - All use @DRAM, consider @SRAM for smaller buffers.
   - acc, mx, sum_val, etc. are already @Stack which is good.
   - Could explore @SRAM for tiled versions.

8. **No tiling/loop reordering** - Kernels are naive implementations.
   - matmul variants could benefit from tiling for cache locality.
   - Consider memory access patterns for GPU/CPU acceleration.

9. **Numerical stability** - 
   - softmax uses expf which could overflow for large inputs.
   - cross_entropy uses log(p + 1e-10) which is good.
   - Consider log-sum-exp trick more explicitly.

10. **Type consistency** - All f64, but f32 might be faster on some hardware.
    - Consider templating or configurable precision.

## Attention Kernel Design (jax.nn.dot_product_attention inspired)

### Current State
- `attn_fwd_fused` (bench_exojit.py:103-150) does everything: rmsnorm → QKV projection → attention → softmax → V weighted sum → output proj → residual
- `attn_bwd_fused` (bench_exojit.py:152-211) handles the backward pass

### Proposed: Extract `dot_product_attention` kernel

```python
@proc
def dot_product_attention(
    S: size,           # seq_len (BLOCK_SIZE)
    H: size,           # n_heads
    D: size,           # head_dim
    out: f64[H, S, D] @ DRAM,
    q: f64[H, S, D] @ DRAM,
    k: f64[H, S, D] @ DRAM,
    v: f64[H, S, D] @ DRAM,
    attn_w: f64[H * S, S] @ DRAM,  # workspace for attention weights
    scale: f64[1] @ DRAM,
):
    # For each head and query position:
    #   1. Compute scores = q @ k.T * scale (with causal mask)
    #   2. Softmax over scores
    #   3. out = softmax(scores) @ v
```

### Design Decisions

1. **In-place vs workspace**: Current code uses `attn_w` as workspace. Keep this pattern.

2. **Causal masking**: Built into the kernel (as JAX does with `is_causal=True`).

3. **Scale**: Pass as parameter (1/sqrt(D)) rather than hardcoding.

4. **Backward**: Would need `dot_product_attention_bwd` that computes:
   - dv = attn.T @ dout
   - dattn = dout @ v.T
   - dlogits = softmax_bwd(dattn) * scale
   - dq = dlogits @ k
   - dk = dlogits.T @ q

5. **Integration path**:
   - Add `dot_product_attention` to exo_kernels.py
   - Add `dot_product_attention_bwd` to exo_kernels.py
   - Refactor `attn_fwd_fused` to call it (lines 124-146 become one call)
   - Refactor `attn_bwd_fused` similarly

6. **Benefits**:
   - Testable in isolation
   - Reusable for multi-layer transformers
   - Matches JAX mental model
   - Cleaner separation of concerns

7. **Open questions**:
   - Should QKV projection be separate kernel or stay fused?
   - Flash Attention style memory optimization (recompute vs materialize attn_w)?
   - Multi-query / grouped-query attention support?
